# Settings Architecture and Split Map

_Last updated 2026-07-18 by GPT._

This document explains the current ownership boundaries inside the Quickshell Settings application. It exists so future structural work does not accidentally move UI presentation and transaction behavior together, regress Apply/Cancel, or rebuild from an older `SettingsWindow.qml` parent.

## Current structure

```text
widgets/Settings/
├── SettingsWindow.qml
├── components/
│   ├── SettingsPendingFooter.qml
│   └── other reusable Settings controls...
└── pages/
    ├── AppearancePage.qml
    ├── DesktopPage.qml
    ├── HyprlandPage.qml
    ├── NotificationsPage.qml
    └── SddmPage.qml
```

The page files own page-specific controls and presentation. `SettingsWindow.qml` remains the application host and still owns the shared staged transaction.

## Rev 20: pending footer extraction

`widgets/Settings/components/SettingsPendingFooter.qml` now owns only the fixed footer presentation:

- pending-change heading and count;
- three-line pending-change list;
- empty-state text;
- `ConfigManager.busy`, `lastError`, and `lastOutput` status display;
- Cancel and Apply buttons;
- button enabled/disabled presentation.

It deliberately does **not** own staged values or persistence.

### Public interface

```qml
SettingsPendingFooter {
    changes: root.changes
    pendingVisibleLines: root.pendingVisibleLines
    onCancelRequested: root.discardStaged()
    onApplyRequested: root.apply()
}
```

The component contract is:

- `changes`: array/list of `{ label, from, to }` objects generated by the window;
- `pendingVisibleLines`: fixed number of visible pending rows;
- `cancelRequested()`: presentation signal only;
- `applyRequested()`: presentation signal only.

`SettingsWindow.qml` remains authoritative for:

- every `staged...` property;
- deriving `changes`;
- `discardStaged()`;
- `apply()`;
- validation;
- transaction ordering;
- calls into `ConfigManager`;
- page navigation and shared popups.

Do not move transaction behavior into `SettingsPendingFooter.qml`. Keeping that component dumb makes it safe to restyle or replace without changing persistence behavior.

## Fixed-height footer behavior

The pending panel intentionally keeps a fixed geometry while changes are staged and unstaged. The list contents change, but the footer and buttons must not jump vertically under the pointer.

Do not replace the fixed list height with content-driven height, and do not hide the pending panel when the list is empty. Those patterns previously caused the page viewport and Apply/Cancel controls to move during editing.

## Troubleshooting Rev 20

### Pending rows do not update

Check that `SettingsWindow.qml` passes `changes: root.changes`. Then verify the staged property appears in the window's `changes` derivation. The footer does not discover changes itself.

### Cancel button does nothing

Check:

```qml
onCancelRequested: root.discardStaged()
```

The footer only emits the signal. The window owns the discard implementation.

### Apply button does nothing

Check:

```qml
onApplyRequested: root.apply()
```

Also inspect `ConfigManager.busy`. Apply is intentionally disabled while a transaction is running.

### Apply remains disabled

The button requires both:

```text
changes.length > 0
ConfigManager.busy === ""
```

A stuck busy state belongs to `ConfigManager` or the active transaction, not to the footer component.

### Status text is missing or stale

`SettingsPendingFooter.qml` imports `qs.services` and reads `ConfigManager` directly for display. Verify the import and the singleton state before changing the footer API.

### Footer or buttons jump when staging changes

Verify that:

- the pending panel remains present when empty;
- `ListView.Layout.preferredHeight` remains based on `pendingVisibleLines`;
- the footer is still outside the scrolling page viewport;
- no page has reabsorbed the Apply/Cancel controls.

## Live-test checklist after future Settings splits

Stage at least one obvious setting on each normal Settings tab, then verify:

1. every staged change appears in the footer;
2. nothing touches disk before Apply;
3. Cancel returns all staged controls to live values;
4. Apply commits all tabs in one transaction;
5. pending rows clear after a successful Apply;
6. status/busy/error text remains visible;
7. the footer stays fixed while the page scrolls;
8. Apply remains disabled during an active `ConfigManager` transaction.

SDDM may require Test mode or a real Apply/reboot to make a staged change visually obvious, so it is not required for the quick multi-tab smoke test.

## Recommended next split

The next meaningful extraction is the staging/transaction controller currently inside `SettingsWindow.qml`. Do that as a separate revision after Rev 20 is committed and used as the exact parent.

A future controller may own staged properties, change derivation, discard, validation, and Apply orchestration. The window should eventually become mostly:

- FloatingWindow lifecycle and preferred sizing;
- navigation/sidebar;
- page hosting;
- shared popup hosting;
- connection of the transaction controller to pages and footer.

Do not combine the transaction-controller extraction with new launcher, wallpaper-picker, or notification features. First preserve behavior under a structural-only revision and live-test it.
