=================================================================
PURPOSE
=================================================================

Implementation and expansion plan for Hyprland animation presets in the
settings window. Option A is complete and live-tested through Rev 39 with
four staged presets and a manager-owned generated/animations.lua.

Unlike SDDM, this ISN'T a separate project — it's a new page (or new
section of the Hyprland page) in the existing settings window,
touching the existing ConfigManager/UserPrefs machinery. The scope
question isn't "where does this live," it's "how much of Hyprland's
animation system actually gets exposed," which is why this plan leads
with that decision instead of a phase list.

=================================================================
WHAT HYPRLAND ACTUALLY SUPPORTS (the ceiling)
=================================================================

Currently in user/look.lua, hand-owned, NOT generated by the shell:

- 5 named bezier curves (easeOutQuint, easeInOutCubic, linear,
  almostLinear, quick) — each 2 control points, same shape as CSS
  cubic-bezier().
- 1 spring curve (easy) — mass/stiffness/dampening.
- 16 animation "leaves" (global, border, windows, windowsIn,
  windowsOut, fadeIn, fadeOut, fade, layers, layersIn, layersOut,
  fadeLayersIn, fadeLayersOut, workspaces, workspacesIn,
  workspacesOut, zoomFactor) — each with enabled/speed/curve-or-spring/
  optional style ("popin 87%", "slide", "fade", etc).

So in principle, yes: literal per-leaf, per-control-point editing is
possible. Whether it's WORTH building at that resolution is the real
question — see the two options below.

=================================================================
THE ARCHITECTURAL CATCH
=================================================================

None of the above is currently a managed file. ConfigManager owns and
generates monitors.lua, appearance.lua, and a handful of UserPrefs-
backed values — user/look.lua is hand-edited and the shell has never
touched it. Exposing ANY of this in the settings window means an
ownership transfer, same category of change SONNET_QUEUE's Q8 (a
single animations.enabled toggle) already flagged as "ask before
touching."

Risk profile, for calibration: a malformed Hyprland config can fail
the WHOLE config reload (not just look ugly), so this needs the same
apply-with-revert-timer pattern ConfigManager already uses for other
generated files (see writeRevertScript/revertRestoreScript) — snapshot
first, write, open a revert window, confirm or auto-revert. Lower risk
than the Displays/monitors page (a bad animation config doesn't black-
screen you), but not zero, and not a plain UserPrefs JSON write either.

=================================================================
TWO WAYS TO SCOPE THIS — PICK ONE BEFORE BUILDING
=================================================================

---- Option A — Presets (recommended starting point) ----

One UserPrefs enum (e.g. "off" / "snappy" / "smooth" / "bouncy"),
each mapping to a hardcoded, pre-tuned table of ALL 16 leaves' speed/
curve/style values. Picking a preset regenerates the whole animations
block as one unit — the user picks a feel, not sixteen individual
knobs.

Pros: small ownership transfer (one generated file, one clear
before/after diff), tiny UI (a single dropdown or segmented row, same
shape as the Theme picker), most of the practical value (most people
want "make it snappier" or "turn it off," not to hand-tune
windowsOut's bezier independently).
Cons: no fine-grained control — if none of the 4 presets feels right,
there's no dial to turn.

---- Option B — Full per-leaf raw editing ----

Real bezier-curve editing (drag control points, or numeric x1/y1/x2/y2
fields) plus per-leaf enabled/speed/curve/style pickers, for all 16
leaves individually. This is the actual ceiling from the section
above.

Pros: total control, nothing left on the table.
Cons: real design work on multiple fronts — a UI for 16 leaves that
doesn't turn into a wall of controls (accordion? only show
"commonly tweaked" leaves by default?), a bigger ownership transfer
(the whole animations+curves block, not one preset value), and more
surface area for something to look subtly wrong after Apply. This is
squarely in "needs a new pattern" territory per the project's parked-
items list — not a reason to avoid it, but a reason not to start here.

Recommendation: build A first, live with it, decide whether B is
actually wanted once there's real usage to judge by. B can layer on
top of A later (e.g. an "Advanced" toggle that unlocks per-leaf
overrides on top of whatever preset is active) rather than replacing
it — worth keeping that door open when A's data shape gets designed.

=================================================================
PHASES (Option A track)
=================================================================

---- Phase 0 — Ownership decision + ask (COMPLETE) ----

Goal: get explicit sign-off before any code, same as Q8 required for
a single key.

- Confirm: does ALL animation config move to a new generated file
  (e.g. hypr/generated/animations.lua, sourced from hyprland.lua same
  as appearance.lua/monitors.lua), with user/look.lua's existing
  animations{}/hl.curve()/hl.animation() calls REMOVED (ownership
  transfer, not a duplicate)? That's the clean option — no two
  sources of truth for the same leaves.
- Confirm the 4 preset names and rough feel wanted (off / snappy /
  smooth / bouncy is a starting guess, not a commitment).
- Confirm current values in look.lua are the "off"/baseline preset's
  source (or a fresh baseline) — don't invent numbers nobody asked
  for.

---- Phase 1 — Preset table + generation script (IMPLEMENTED, UNTESTED) ----

Goal: a ConfigManager-generated animations.lua that reproduces
CURRENT behavior exactly under the default preset — no visual change
until someone picks a different one.

- Hardcode 4 preset tables (leaf -> {enabled, speed, curve/spring,
  style}) as a ConfigManager readonly property, same shape as the
  hyprGenScript template pattern used for monitors.lua/appearance.lua.
- Generation script writes the FULL animations{}/hl.curve()/
  hl.animation() block matching look.lua's current syntax exactly.
- Test offline against a sandbox tree first (same discipline as
  ConfigManager Phase 1's original build — extract the sh, run it
  against copies, diff the output against current look.lua for the
  baseline preset BEFORE it ever touches a real config).

---- Phase 2 — UserPrefs + settings window UI (IMPLEMENTED, UNTESTED) ----

Goal: a working staged/Apply control, same shape as every other
Hyprland-page setting.

- New UserPrefs.hyprAnimationPreset (string enum), default matching
  whatever Phase 0 confirmed as baseline.
- Settings window: single dropdown or segmented row (OptionPickerRow
  is probably right here — 4 options, not 14 like the wallpaper
  transition type) on the Hyprland page.
- Apply uses the file-with-revert-timer pattern (see ARCHITECTURAL
  CATCH above), NOT a plain UserPrefs JSON write — this is the one
  place this phase differs from a typical settings-window addition.

---- Phase 3 — Live test (COMPLETE) ----

Result: all four presets were applied on the real machine, repeated rapid
Applies were tested, and the final normal-reload path did not crash Hyprland.
Malformed-generation experiments exposed API and quoting failures during
Revs 30–38; those are documented in PROBLEMS_AND_FIXES.md. The final design
does not use a countdown revert window or full Lua-context reset.

=================================================================
OUT OF SCOPE FOR NOW
=================================================================

- Full per-leaf raw editing (Option B) — see above; a later phase at
  earliest, not part of this plan.
- Bezier curve visual editor (drag control points) — only relevant if
  Option B ever happens; presets don't need it.
- Anything reading/parsing the CURRENT look.lua animation values at
  runtime — Phase 1 hardcodes preset tables instead. Parsing arbitrary
  Lua back into structured data is a much harder, separate problem
  nobody's asked for and isn't needed for presets.

=================================================================
FINAL IMPLEMENTATION STATUS — REVS 30–39
=================================================================

- Option A is complete: Off, Snappy, Smooth, Bouncy.
- Smooth preserves the old user/look.lua baseline.
- Animation ownership lives in generated/animations.lua.
- The migration script removes the old hand-owned animation block and ensures
  generated animations load after user.look.
- Hyprland Lua speed values are durations in deciseconds; lower values are faster.
- Generated files use the global hl API directly. Do not require("hyprlang").
- Hyprland color values are strings (for example "rgba(33ccffee)").
- Solid and gradient appearance code is emitted through direct heredoc branches,
  not nested shell-string assembly.
- Safe final Apply path: write appearance.lua, write animations.lua, then run
  one ordinary hyprctl reload.
- Never use hyprctl reload full-reset from Settings Apply. Repeated full resets
  crashed Hyprland even with debounce/cooldown logic.
- Never use hyprctl eval/dofile for this feature. The experiment was removed.
- UI Profiles includes the persisted animation preset and reuses the normal
  Hyprland regeneration path after restore.

Future expansion remains Option B only: advanced per-leaf editing and optional
curve controls layered on top of the stable preset system.
